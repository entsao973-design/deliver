# 智慧拍照第 1、2 步驟設計

日期：2026-06-16

範圍只包含：
1. Excel 匯入後保存單據地址。
2. 後端將地址轉成 GPS 經緯度，並保存定位狀態。

不包含本階段：
- 司機端「智慧拍照」按鈕。
- 手機定位與 300 公尺匹配。
- 語音提示。
- 拍照後自動繼續下一筆。

## 目標

匯入 Excel 後，每筆配送單據需要保留地址，並盡可能取得可用的 GPS 座標。後續司機端智慧拍照只需要讀取同車號、同日期、未達交且已有 GPS 的單據，不需要在手機端做地址解析。

同一地址可能有多張單據，因此地址轉 GPS 必須能重用結果，避免重複呼叫外部服務、增加成本與等待時間。

## 現況

- Excel 測試資料已有「地址」欄，位置為第 C 欄。
- `delivery_app/excel_importer.py` 目前讀取客戶、公司、出貨單號等欄位，但沒有保存地址。
- JSON repository 與 SQL Server schema 目前沒有地址、經緯度、定位狀態欄位。
- 目前正式設定 `initialize_schema=false`，因此 SQL Server 欄位新增不應依賴每次啟動自動建表或改表。

## 建議方案

採用「單據欄位 + 地址定位快取」雙層設計。

配送單據本身保存地址與 GPS 結果，方便司機端直接取得：
- `address`
- `geocode_lat`
- `geocode_lng`
- `geocode_status`
- `geocode_updated_at`

另外建立地址定位快取，以標準化地址作為 key：
- `normalized_address`
- `provider`
- `provider_place_id`
- `lat`
- `lng`
- `status`
- `error_message`
- `updated_at`

理由：
- 多張單據同地址時，只需轉換一次。
- 司機端查詢單據時不必再 join 或重新呼叫外部服務。
- 未來若更換 Google、TGOS 或其他 geocoding 服務，只要換 provider 實作。
- 地址定位結果可被管理端檢查與修正。

## 第 1 步：地址入庫

Excel 匯入器新增讀取第 C 欄地址：
- 空白地址保存為空字串。
- 合併列沿用目前「客戶名稱、公司」的處理模式：同一序號跨多列時，若地址欄空白，沿用上一個非空地址。
- delivery record 新增 `address` 欄位。

匯入更新規則：
- 新單據：保存地址，若地址非空，`geocode_status = "pending"`。
- 既有未達交單據：若地址變更，更新地址，清空舊座標，`geocode_status = "pending"`。
- 既有未達交單據：若地址未變，保留既有座標與定位狀態。
- 已達交單據：維持目前鎖定邏輯，不因重新匯入覆蓋已達交內容。
- 空白地址：`geocode_status = "empty"`，座標為空。

公開 API 回傳單據時建議包含：
- `address`
- `geocode_lat`
- `geocode_lng`
- `geocode_status`

司機卡片是否顯示地址可先不做，避免第 1、2 步擴大 UI 範圍。

## 第 2 步：地址轉 GPS

新增後端 geocoding 模組，例如 `delivery_app/geocoding.py`。

核心介面：
- 輸入：原始地址。
- 輸出：標準化地址、緯度、經度、狀態、供應商名稱、供應商地點 ID、錯誤訊息。

狀態建議：
- `pending`：等待定位。
- `success`：已取得可用經緯度。
- `empty`：地址空白，不處理。
- `no_result`：供應商查無結果。
- `ambiguous`：結果不夠明確，不自動用於智慧拍照。
- `failed`：呼叫失敗、逾時或配額問題。
- `manual`：未來管理端人工修正座標。

定位流程：
1. Excel 匯入完成後，找出同日期或本次匯入的 `pending` 地址。
2. 先查地址定位快取。
3. 快取有 `success` 結果時，直接寫回所有同地址單據。
4. 快取沒有結果時，由後端呼叫 geocoding provider。
5. 將結果寫入快取，再同步寫回對應單據。
6. 失敗時保存狀態與錯誤訊息，不阻止 Excel 匯入成功。

## 執行方式建議

建議採「匯入後背景處理 + 管理端可手動重跑」。

匯入 Excel API 不應等待所有地址定位完成，否則大批資料或外部 API 慢速時，管理端會誤以為上傳失敗。

流程：
- `匯入 Excel` 完成後立即回傳匯入結果。
- 伺服器背景執行 geocoding。
- 管理端後續可看到定位統計，例如成功、待處理、失敗、地址空白。
- 若外部服務暫時失敗，可由管理端按「重新定位地址」重跑失敗或待處理資料。

## 設定設計

`config.json` 建議新增：

```json
{
  "geocoding": {
    "enabled": false,
    "provider": "google",
    "api_key_env": "GOOGLE_GEOCODING_API_KEY",
    "country_hint": "台灣",
    "timeout_seconds": 5,
    "rate_limit_per_second": 5
  }
}
```

注意：
- API key 優先用環境變數，不直接寫在 `config.json`。
- `enabled=false` 時只保存地址，不呼叫外部服務。
- 供應商實作必須在後端，不可把 API key 暴露給前端。

## SQL Server schema 設計

`dbo.deliveries` 建議新增欄位：

```sql
address nvarchar(1000) NOT NULL CONSTRAINT DF_deliveries_address DEFAULT N'',
geocode_lat decimal(9,6) NULL,
geocode_lng decimal(9,6) NULL,
geocode_status nvarchar(20) NOT NULL CONSTRAINT DF_deliveries_geocode_status DEFAULT N'pending',
geocode_provider nvarchar(50) NULL,
geocode_place_id nvarchar(255) NULL,
geocode_updated_at datetime2(0) NULL,
geocode_error nvarchar(500) NULL,
normalized_address nvarchar(1000) NULL
```

另建快取表：

```sql
CREATE TABLE dbo.address_geocode_cache (
    normalized_address nvarchar(1000) NOT NULL CONSTRAINT PK_address_geocode_cache PRIMARY KEY,
    address nvarchar(1000) NOT NULL,
    provider nvarchar(50) NOT NULL,
    provider_place_id nvarchar(255) NULL,
    lat decimal(9,6) NULL,
    lng decimal(9,6) NULL,
    status nvarchar(20) NOT NULL,
    error_message nvarchar(500) NULL,
    updated_at datetime2(0) NOT NULL
);
```

因正式環境 `initialize_schema=false`，建議做一次性遷移腳本，而不是啟動時自動改表。

## JSON repository 相容設計

JSON 模式也要保存相同欄位，方便本機測試與既有單元測試：
- delivery record 直接新增上述欄位。
- 地址快取可放在資料檔 top-level，例如 `geocode_cache`。

若舊資料沒有新欄位，讀取時用預設值補齊：
- `address = ""`
- `geocode_status = "empty"` 或 `pending`，依 address 是否空白判斷。
- 經緯度為 `null`。

## 錯誤處理

Excel 匯入不應因 geocoding 失敗而整體失敗。

以下情境只更新定位狀態：
- 地址空白：`empty`
- 查無結果：`no_result`
- 結果不明確：`ambiguous`
- 供應商逾時或網路錯誤：`failed`
- API key 未設定：若 `enabled=true` 則記為 `failed`，錯誤訊息為設定缺漏。

## 測試設計

建議新增或更新測試：
- Excel 匯入會保存第 C 欄地址。
- 合併列地址可沿用上一個非空地址。
- 未達交單據地址變更時，座標會清空並重新標記 `pending`。
- 地址未變時，既有 `success` 座標不被清掉。
- 空白地址標記為 `empty`。
- SQL Server schema 欄位與快取表存在。
- geocoding provider 使用 fake provider 測試，不在測試中呼叫真實外部 API。
- geocoding 失敗不會讓 Excel 匯入失敗。

## 待確認決策

1. 第一個 geocoding provider 是否使用 Google Geocoding API。
2. 是否接受匯入後背景自動定位；若想完全控管 API 成本，可改成管理端按鈕手動定位。
3. 管理端是否需要在第 2 步同時顯示定位統計；若不做，後台不容易知道哪些地址定位失敗。

## 建議結論

第 1、2 步可以先做為後端能力，不碰司機操作介面。

推薦採用：
- 第 1 步：Excel 匯入保存地址。
- 第 2 步：建立 geocoding 欄位、快取表、provider 介面。
- geocoding 先做背景處理，失敗不阻止匯入。
- 管理端至少顯示定位統計，手動修正可以放後續階段。
