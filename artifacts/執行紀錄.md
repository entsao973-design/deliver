### 智慧拍照 GPS 地址匹配功能可行性評估

- 使用者提出功能：Excel 匯入後把單據地址轉成 GPS 點；物流士按「智慧拍照」後，使用手機當下定位與單據地址 GPS 比距離，300 公尺內自動匹配單據並進入拍照流程。
- 查核現況：
  - 測試 Excel 欄位已有「地址」欄。
  - `delivery_app/excel_importer.py` 目前未把地址寫入 delivery record。
  - JSON repository 與 SQL Server schema 目前沒有 address / latitude / longitude / geocode 狀態欄位。
  - 前端司機頁已有拍照、離線暫存、照片上傳流程，可被智慧拍照流程重用。
- 初步判斷：
  - 功能可行，但不是純前端功能，需要資料匯入、資料庫欄位、地址轉 GPS、司機端定位匹配、UI/語音提示一起設計。
  - 因 `config.json` 目前正式設定為 `initialize_schema=false`，SQL Server 新欄位不應靠每次啟動自動新增，建議做一次性遷移或明確管理端升級動作。
- 主要風險：
  - Android 手機取得定位通常需要 HTTPS secure context；若手機用區網 HTTP 連到 PC，可能拿不到定位權限。
  - 地址轉 GPS 可能不準，尤其地址格式不完整、同名道路、工業區、園區、大樓入口與收貨口不同時。
  - Google Geocoding 等外部服務有計費、配額與使用條款；是否能長期保存經緯度需依選定服務確認。
  - 語音提示可用 Web Speech / SpeechSynthesis，但不同 Android 裝置的中文語音效果需實機確認。
  - 300 公尺門檻可行，但需同時參考手機定位 accuracy，避免定位誤差太大時自動選錯單。
- 建議分階段：
  1. 匯入並保存地址欄位。
  2. 加入地址轉 GPS 與 geocode 狀態，不影響原拍照流程。
  3. 司機頁新增「智慧拍照」按鈕，只做 300 公尺內匹配與選單。
  4. 加入語音提示與拍完後繼續智慧拍照。
  5. 加入管理端校正/檢查地址定位結果。

### Android 檢視所有照片單指滑動捲動修正

- 問題現象：
  - 物流士端勾選「檢視所有照片」後，若手指起始位置剛好在照片區，Android 上單指上下滑動不會捲動配送清單。
  - iOS 上較不明顯或不會發生。
- 根因判斷：
  - `.inline-photo-viewport` 原本預設 `overscroll-behavior: contain`，照片區本身又是 `overflow: hidden` 的固定高度容器。
  - Android Chrome 會較嚴格遵守 `overscroll-behavior` 與 Pointer Events，可能把滑動留在照片區，不再交給外層清單。
  - `photo-viewer.js` 原本在單指 `pointerdown` 時就呼叫 `setPointerCapture(...)`，即使照片尚未放大，也會讓 Android 更容易把後續滑動視為照片元件內部操作。
  - iOS Safari 對 `overscroll-behavior`、`touch-action`、Pointer Events 的實作與 Android Chrome 不完全相同，未放大時較常把垂直滑動交回頁面捲動，所以 iOS 看起來正常。
- TDD 驗證：
  - `tests/photo_viewer.test.js`

### 匯入 xlsm 時不再保留 VBA archive

  - 卡點位於 `openpyxl.load_workbook(...)`。
- 判斷：
  - 本系統匯入 Excel 只讀取儲存格資料，不會修改或另存原 Excel。
  - 因此 `.xlsm` 匯入時不需要 `keep_vba=True`。
  - 保留 VBA archive 會讓 openpyxl 走更重的巨集封存讀取路徑，可能增加工作排程 `SYSTEM` 帳號下卡住或原生/壓縮檔讀取異常的風險。

  - `delivery_app/excel_importer.py`
    - `openpyxl.load_workbook(...)` 的 `keep_vba` 由依副檔名判斷改為固定 `False`。
    - 保留 `read_only=False` 與 `data_only=True`，不改既有欄位讀取與合併儲存格邏輯。
